# Plan: Transform `flog-workout` into a fullstack pnpm monorepo

## Context

`flog-workout` today is a Vite + React 19 + TS SPA with TanStack Router and Tailwind/DaisyUI. All data is hardcoded in [src/dummyData.ts](src/dummyData.ts) (6 workouts, 12 exercises). The existing GH Actions workflow `aws s3 sync`s the build to a bucket and invalidates CloudFront ([.github/workflows/main.yml](.github/workflows/main.yml)).

The goal is to turn it into a real fullstack app: keep the existing frontend, add a Hono + Postgres backend, and deploy on AWS. The current S3 deploy is allowed to break.

Two decisions are deliberately deferred to Phase F (when they matter):
- **DB flavor:** classic RDS Postgres vs Aurora Serverless v2 (Postgres-compatible). Prisma works on both — same code.
- **IaC tool:** SST v3 vs Terraform. The choice changes Phase F's shape but not Phases A–E.
- **Frontend hosting (decided):** AWS Amplify Hosting. Replaces the hand-rolled S3 sync with Amplify's native pnpm pipeline + GitHub integration.

## Working style

This migration is a learning exercise — you drive the keyboard, I assist where boilerplate would slow learning more than it teaches. Each phase below has a **Working notes** line marking parts as **DIY** (hand-code for the learning) or **delegate-friendly** (mechanical typing, fine to ask Claude to scaffold and then read what was produced). When in doubt, default to DIY — even small files like `package.json` are worth typing once.

## Target architecture

- **Frontend:** Vite SPA → AWS Amplify Hosting (auto-builds & deploys on push to `main`).
- **Backend:** Hono on AWS Lambda behind API Gateway (HTTP API), in a VPC.
- **DB:** Postgres on RDS (flavor TBD Phase F), in the same VPC as the Lambda.
- **Type-safe link:** Hono's `hc` RPC client + a `packages/shared` Zod-schemas package.
- **Local dev:** Postgres in Docker; web on `:3000`, api on `:3001`; `pnpm dev` runs both in parallel.

**Why this split:** `apps/web` and `apps/api` have disjoint runtime deps (React vs Node) and ship to different targets. `packages/shared` exists so the API's Zod validators and the SPA's form types are *literally the same source of truth*. No fourth `eslint-config`/`tsconfig` package — overkill at this scale.

**Package names:** `@flog/web`, `@flog/api`, `@flog/shared`. `pnpm-workspace.yaml` adds `packages: [apps/*, packages/*]` (preserve the existing `minimumReleaseAgeExclude` for `@tanstack/*`).

## Type / contract sharing

Use Hono's end-to-end RPC. No OpenAPI codegen, no manual SDK.

- **`@flog/shared`** owns Zod schemas (`WorkoutSchema`, `ExerciseSchema`, `CreateWorkoutSchema`, …) and exports inferred types via `z.infer`. Pure TS, no build step (`"exports": { ".": "./src/index.ts" }`). The web bundler compiles it; the api bundler compiles it.
- **`@flog/api`** uses those schemas with `@hono/zod-validator` on routes, and exports `type AppType = typeof app`.
- **`@flog/web`** does `import type { AppType } from "@flog/api"` (type-only), constructs `hc<AppType>(import.meta.env.VITE_API_URL)`, wraps it in TanStack Query.

Enforce **type-only** import of `@flog/api` from web via ESLint `no-restricted-imports` — a value import would drag the whole API into the SPA bundle.

## Backend architecture

**App split.** One `app.ts` builds the Hono app, shared by `index.ts` (local node-server) and `lambda.ts` (Lambda handler via `hono/aws-lambda`'s `handle()`). Same code, different entry.

```ts
// apps/api/src/app.ts (sketch)
const app = new Hono()
  .use("*", logger())
  .use("*", cors({ origin: process.env.CORS_ORIGIN ?? "http://localhost:3000" }))
  .route("/api/workouts", workouts)
  .route("/api/exercises", exercises)
  .get("/health", (c) => c.json({ ok: true }))
export default app
export type AppType = typeof app
```

```ts
// apps/api/src/routes/workouts.ts (sketch)
import { prisma } from "../db/client"
export default new Hono()
  .get("/", async (c) =>
    c.json(await prisma.workout.findMany({ include: { exercises: { include: { exercise: true } } } }))
  )
  .post("/", zValidator("json", CreateWorkoutSchema), async (c) => {
    const row = await prisma.workout.create({ data: c.req.valid("json") })
    return c.json(row, 201)
  })
```

**DB layer: Prisma.** You picked Prisma — fair call for a learning project. The schema language is its own thing worth learning, the docs are excellent, and Prisma Studio is a great DB browser. The trade-offs to know:

- **Lambda binary engine.** Prisma's default query engine is a native binary that's bundled into the Lambda package. Two paths in Phase F:
  - **Default engine + `binaryTargets`.** Set `binaryTargets = ["native", "rhel-openssl-3.0.x"]` in `schema.prisma` for AWS Lambda Node 20+. Bundle is bigger (~15–25 MB), cold-start adds ~100–200 ms. Workable for a personal app.
  - **Driver Adapters (Prisma 5.10+).** Use `@prisma/adapter-pg` with the `pg` driver to drop the binary entirely — same Prisma client API, no native code in the bundle. Pick this in Phase F if cold-start latency matters or bundle size becomes a problem. Phases A–E don't care: local dev runs the default engine fine.
- **Connection pooling.** Lambda concurrency × Prisma's per-instance connections will exhaust Postgres's `max_connections` quickly. In Phase F, fix with either RDS Proxy (managed by your IaC) or Prisma Accelerate (managed SaaS, paid). Local dev is unaffected.

**Schema (3 models in Prisma terms).** `Workout`, `Exercise`, `WorkoutExercise` join model. Current dummy data shows exercises are reusable across workouts — model the many-to-many from day one rather than denormalized JSON. The API's `GET /workouts/:id` uses Prisma's nested `include` so the response keeps today's `{ ...workout, exercises: Exercise[] }` shape (with a small `map` to flatten the join through the relation).

**Migrations:** `pnpm prisma migrate dev --name <description>` creates a migration SQL file in `apps/api/prisma/migrations/` AND applies it locally. In prod, `pnpm prisma migrate deploy` (no schema diffing, just applies pending migrations) runs as a deploy step.

**Seeding:** `apps/api/prisma/seed.ts` ports the existing `dummyData.ts` shape verbatim, idempotent via `upsert`. Wire it via `package.json` `prisma.seed` so `pnpm prisma db seed` runs it.

Amplify connects to the GitHub repo, watches `main`, runs the `amplify.yml` build spec. The current S3-sync workflow is no longer needed for frontend.

**API URL plumbing (frontend ⇄ backend split deploy).** The backend deploy publishes the API Gateway URL to AWS Systems Manager Parameter Store at `/flog/prod/api-url`. Amplify Hosting environment variables can reference SSM parameters — set `VITE_API_URL` in the Amplify console to point at that param. Result: deploy backend first → SSM updates → trigger Amplify rebuild → SPA picks up new URL. For PR previews, do similarly with stage-suffixed param names.

A simpler alternative (if SSM-from-Amplify proves fiddly): assign a stable custom domain to API Gateway (e.g. `api.flog-workout.example`) and hardcode it in Amplify env vars. This decouples deploys but needs a domain.

## Backend deploy (IaC choice deferred to Phase F)

Phases A–E are IaC-agnostic — no infra needed until Phase F. When we get there, the choice is:

- **SST v3** — purpose-built for fullstack TS on AWS. One config wires Lambda + API Gateway + RDS + per-PR preview stages. The cost: a non-trivial dependency that owns your deploy.
- **Terraform** — vendor-neutral, mature, you already mentioned familiarity. More boilerplate; no "scaffold a Lambda from a TS handler" magic — you bring your own bundler step (esbuild) and write the function/API/RDS/SG/VPC HCL.

Both fit. The decision is "how much do we want the IaC tool to do for us vs how much control / portability do we want?" Pick at the top of Phase F based on appetite at that point.

Either way, the Phase F shape:

1. Provision a small VPC (2 AZs, 2 private subnets, 1 public subnet).
2. Provision Postgres in the VPC. Flavor decision (Aurora Serverless v2 vs RDS t4g.micro) made here. Prisma code is identical.
3. Bundle the Lambda handler. With Prisma's default engine: include the `prisma/` schema + the matching `query-engine-rhel-openssl-3.0.x` binary in the bundle (Prisma's `@prisma/client` post-install copies it; bundlers may need a manual copy step). With Prisma Driver Adapters: bundle just JS, mark Node built-ins as external.
4. Provision Lambda in the VPC, env vars include `DATABASE_URL` (from Secrets Manager) and `CORS_ORIGIN` (Amplify domain).
5. Provision an HTTP API (API Gateway v2) → Lambda integration.
6. Add RDS Proxy (or commit to Prisma Accelerate) in front of the DB to handle Lambda concurrency.
7. Publish the API URL to SSM Parameter Store at `/flog/prod/api-url`.
8. Run `prisma migrate deploy` against the prod DB as a deploy step.

**Cost note:** Aurora Serverless v2 added scale-to-zero in late 2024 which makes it cheap for intermittent personal use; classic RDS t4g.micro is always-on but predictable. Verify current pricing in eu-central-1 at the time of Phase F before committing — these numbers move.

## CI / CD

Two pipelines after migration:

- **Frontend (Amplify):** Amplify watches `main` (and PRs, optionally) — no GH Actions step. PR previews via Amplify's preview environments.
- **Backend (`.github/workflows/deploy-backend.yml`):** triggered on push to `main` when `apps/api/**`, `packages/shared/**`, or IaC config changes. Steps: install pnpm → install deps → assume AWS role via OIDC → run IaC (sst deploy / terraform apply) → run `prisma migrate deploy` against prod.

**Switch AWS auth to OIDC** in the new workflow (replacing today's long-lived `AWS_ACCESS_KEY_ID`). One-time IAM role + GitHub OIDC trust policy setup; ships with this migration.

## Risks / gotchas

- **TanStack Router code-gen.** The `tanstackRouter` Vite plugin generates [src/routeTree.gen.ts](src/routeTree.gen.ts) based on Vite's root. After moving to `apps/web/`, delete `.tanstack/` so it regenerates clean. ESLint's `globalIgnores` for the gen file must be updated to the new relative path.
- **`verbatimModuleSyntax: true`** is on in the current tsconfig. The `AppType` import in `apps/web` MUST be `import type` — otherwise Vite tries to bundle the whole API. Add an ESLint rule to enforce.
- **Workspace TS resolution.** For `@flog/shared` to work without a build step, set `"exports": { ".": "./src/index.ts" }` and `"types": "./src/index.ts"` in its package.json. No `tsconfig` paths needed.
- **Dummy data → seed mapping.** [src/dummyData.ts](src/dummyData.ts) references exercises by array *index* (e.g. `dummyExercises[1]`), not by id. The seed script must build the join table by mapping array positions → upserted IDs (upsert by name to stay idempotent), not by id field.
- **Direct mutation in $workoutId.tsx.** Today's flow mutates `dummyExercises.push(...)` and `selectedWorkout.exercises.push(...)` directly. Phase E rewrite must replace these with a real mutation + cache invalidation, not just swap a query in.
- **Prisma binary engine on Lambda.** The query engine is a native binary that must match the Lambda runtime. Set `binaryTargets = ["native", "rhel-openssl-3.0.x"]` in `schema.prisma` (Lambda Node 20+ uses Amazon Linux 2023 / OpenSSL 3). Bundlers must include the `query-engine-rhel-openssl-3.0.x` file from `node_modules/.prisma/client/` in the deployment package — common bundler footgun. Alternative: switch to Prisma Driver Adapters (`@prisma/adapter-pg`) to skip the binary.
- **Prisma client placement in `node_modules`.** The generated client lives at `node_modules/.prisma/client/`. With pnpm's symlinked `node_modules` and bundlers like esbuild, you may need to use `prisma.client.output` config to put the generated client in a deterministic path inside the package — important for Lambda packaging. Document the chosen layout in Phase F.
- **Lambda + Postgres connection storms.** Without RDS Proxy or Prisma Accelerate, Lambda concurrency × per-instance Prisma connections will exhaust `max_connections`. Don't ship Phase F without one of them.
- **Lambda bundling target.** Ship `"type": "module"` in `apps/api/package.json` and target Node 22 ESM. Bundle size with the Prisma engine binary will be larger than typical (~15–25 MB compressed) — within Lambda's 250 MB unzipped limit but worth tracking.
- **Env-specific API URLs.** Don't hardcode in source. Don't ship a runtime config file. Use `VITE_API_URL` baked at build time — locally from `.env`, in Amplify from console env vars (sourced from SSM).
- **DB password / secrets.** Locally: `apps/api/.env` (gitignored — Prisma reads it natively). Prod: AWS Secrets Manager, Lambda reads `DATABASE_URL` injected by IaC, raw secret never appears in logs or git.
- **CORS.** API allows `localhost:3000` in dev, Amplify domain in prod, both via `CORS_ORIGIN` env var.
- **Cost & flavor cliffs.** Both Aurora Serverless v2 idle ACU pricing and t4g.micro hourly pricing have changed in the last 18 months. Re-check eu-central-1 pricing at the time of Phase F before committing.
- **OIDC migration is its own thing.** Switching AWS auth from long-lived keys to OIDC requires a one-time AWS IAM role + GitHub trust policy. Pair with Phase G; don't try it earlier just to get it done.
